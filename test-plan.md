# 餐廳外送系統 - 測試計畫與規格書 (Test Plan & Specification)

本文件依據專案的需求規格書、系統架構設計書、資料模型與開發待辦事項，定義系統的整體測試策略與測試案例，作為系統上線與驗收之依據。測試案例已優先涵蓋登入、餐點瀏覽、購物車、建立訂單、訂單金額計算與訂單狀態更新。

---

## 1. 測試目標、範圍與非測試範圍
*   **測試目標**：驗證餐廳外送系統之前端、後端與資料庫之功能正確性、交易計算邏輯、安全防護與訂單狀態轉移，確保系統於本地 SQLite 與正式 Render PostgreSQL 下皆能穩定運行。
*   **測試範圍**：
    *   消費者、餐廳、外送員與管理者的帳戶管理與 JWT 驗證。
    *   餐廳與選單瀏覽、選餐購物車、金額計算與付款規則校驗。
    *   外送與到店自取之下單流程、外送員自由接單、訂單狀態流轉（狀態機）。
    *   管理者之餐廳營收統計與系統帳戶控制。
    *   API 異常防禦、敏感資訊保護與輸入防護。
*   **非測試範圍 (Out of Scope)**：
    *   真實金流信用卡扣款（此階段僅測試後端 Mock/模擬金流 Token 授權）。
    *   地圖即時 GPS 精準定位軌跡追蹤。

---

## 2. 測試策略
*   **單元測試 (Unit Test)**：
    *   後端：針對 `security.py` 的雜湊與 Token 邏輯、`database.py` 的連線機制、以及 Pydantic 欄位驗證模型進行獨立單元測試。
*   **整合測試 (Integration Test)**：
    *   驗證 API 路由與資料庫 ORM 之間讀寫的正確性，確保 SQLite 與 PostgreSQL 在同等 SQL 語意下表現一致。
*   **API 測試**：
    *   涵蓋所有 Auth、Menu、Orders、Admin 的 RESTful 端點，測試成功與各類 HTTP 異常狀態碼。
*   **前端元件測試 (Frontend Component Test)**：
    *   測試 React 元件在不同狀態（Loading、Error、正常資料）下的渲染情形，與表單輸入邊界值。
*   **端對端測試 (E2E Test)**：
    *   模擬完整使用者故事：消費者登入 -> 選餐入購物車 -> 結帳下單 -> 餐廳接單 -> 外送員接單與配送 -> 送達完成。
*   **手動驗收測試 (Manual User Acceptance Test)**：
    *   由開發者與使用者於部署至 Staging/Staging 伺服器後，進行介面與業務功能的手動測試。

---

## 3. 測試環境與測試資料規則
*   **本機開發與測試環境 (Local)**：
    *   資料庫使用 SQLite (`sqlite:///./delivery.db`)。
    *   每次執行整合/單元測試前，皆須自動刪除並重建資料庫表，並導入測試 Seed 資料（乾淨的使用者與餐廳測試資料）。
*   **正式生產/預備環境 (Production/Staging - Render.com)**：
    *   資料庫使用 Render PostgreSQL。
    *   嚴禁於此環境下執行破壞性的重設（Drop Tables）指令。
    *   測試資料必須使用特定的 Test 標籤帳號，且測試後需能安全地被軟刪除或隔離，不可與正式營收統計數據混淆。

## 5. 技術棧測試重點 (Tech Stack Testing Focus)

### 5.1 FastAPI 後端 API 測試重點
後端 API 必須確保在各種輸入下皆能回傳標準的 HTTP 狀態碼：
*   **成功請求 (200 OK / 201 Created)**：驗證 Response Body 的 JSON 格式與 Schema 完全一致，不含有多餘或缺漏的欄位。
*   **資料驗證失敗 (422 Unprocessable Entity)**：傳入格式錯誤的 JSON、缺少必填欄位或不合規的欄位值時，應自動被 Pydantic 截獲並回傳詳細欄位錯誤說明。
*   **未授權與權限不足 (401 Unauthorized / 403 Forbidden)**：
    *   未帶 JWT Token 存取保護 API 時，應回傳 401。
    *   消費者嘗試操作外送員或餐廳 API 時，應回傳 403。
*   **找不到資源 (404 Not Found)**：查詢不存在的訂單、餐廳或餐點時，應回傳 404。
*   **伺服器錯誤 (500 Internal Server Error)**：任何未捕獲的例外異常應返回 500，且不得直接將完整的 Traceback 錯誤細節回傳給前端，以避免洩漏系統資訊。

### 5.2 React 前端測試重點
*   **畫面呈現**：在元件初次渲染、載入中（Loading Spinner）、無資料（Empty State）時，畫面的呈現必須美觀且佈局不跑版。
*   **使用者互動**：表單即時防呆（例如：輸入欄位字數不夠時，送出按鈕維持 disabled 狀態）。
*   **錯誤提示**：當後端回傳 400（例如「帳號已被使用」）時，前端需以紅字或 Toast 提示使用者，不可使畫面卡死或毫無反應。
*   **API 串接整合**：使用 Axios 攔截器（Interceptors）自動在 Request Headers 注入 JWT Token，並能處理 Token 過期時自動導頁至登入頁。

### 5.3 資料庫測試重點
*   **欄位限制**：使用者帳號與信箱的唯一性（Unique Constraint）限制驗證。
*   **關聯完整性**：驗證當消費者刪除帳戶時，不應造成外送訂單遺失（ON DELETE RESTRICT）；當購物車刪除時，其項目必須自動連帶清除（ON DELETE CASCADE）。
*   **遷移測試 (Migration)**：透過 Alembic 建立與更新資料庫 Schema 時，必須確保在 SQLite 與 PostgreSQL 下皆能無痛運行升級（Upgrade）與降級（Downgrade）。

### 5.4 安全性測試 (Security Testing)
*   **憑證不洩漏**：檢視所有提交的程式碼、環境變數範本（.env.example），確認無任何明文 Secret。確認前端 React 原始碼與編譯出的 Static Files 中，無硬編碼的敏感密鑰。
*   **身份與權限校驗**：模擬篡改 JWT Token（修改簽章或 Payload 角色）嘗試打 API，確認伺服器能辨識並予以拒絕。
*   **防範注入攻擊**：針對輸入欄位（特別是搜尋、文字描述）傳入引號等 SQL 特殊字元，確保 SQLAlchemy ORM 已預防 SQL 注入，且 Pydantic 處理好字串轉義防範 XSS 攻擊。

## 6. 缺陷回報格式與測試完成標準
*   **缺陷回報格式 (Bug Report Template)**：
    *   **缺陷標題**：簡短描述問題（如：消費者於外送結帳時選現金付款未被系統擋下）
    *   **重現步驟**：步驟 1, 步驟 2, ...
    *   **環境資訊**：作業系統、瀏覽器、測試環境（Local / Staging）
    *   **預期結果**：應跳出錯誤提示阻擋該筆下單
    *   **實際結果**：訂單建立成功，資料庫 status 顯示 created
    *   **錯誤日誌/截圖**：貼上 API Log 或 Console 錯誤資訊
*   **測試完成標準 (Sign-off Criteria)**：
    *   所有 P0 核心測試案例（TC-001 ~ TC-006）皆執行通過。
    *   無任何 P0/P1 等級的未修復缺陷（Bugs）。
    *   測試案例整體覆蓋率達到 100%（針對已確認的 MVP 功能）。

## 7. 待確認事項 (Pending Issues)
*   **模擬刷卡異常流程測試**：若信用卡授權失敗、扣款逾時等異常金流狀態，系統目前的重試與狀態轉移規則尚未定義。
*   **取消訂單的系統斷點**：如果外送員已取餐且在配送途中，消費者或餐廳是否能強制取消訂單？其衍生的賠償/退款業務測試流程尚待釐清。
*   **併發訂單的壓力測試界限**：大量外送員在同一秒發送請求承接同一筆訂單時，系統併發處理與防搶單機制的測試基準（QPS 要求）有待明確。
