## MVP 開發階段任務表

### 階段 1：使用者註冊 / 登入 (User Authentication)
*   **優先級**：P0 (最高)
*   **相依關係**：無
*   **狀態**：`[/]`
*   **任務清單**：
    - [x] **Task 1.1: 資料庫基本使用者模型設計**
      - *驗收條件*：完成 users 表 Schema 設計，包含密碼 `bcrypt` 雜湊。
    - [x] **Task 1.2: 後端註冊與登入 API 開發**
      - *驗收條件*：包含 `POST /api/auth/register` 與 `POST /api/auth/login`，登入成功回傳 JWT Token，錯誤密碼與重複帳號需回傳對應錯誤碼。
    - [ ] **Task 1.3: 前端登入/註冊畫面實作**
      - *驗收條件*：實作登入與註冊表單，能發送請求並將 Token 儲存在前端狀態中。

### 階段 2：餐廳與餐點瀏覽 (Restaurant & Menu Browsing)
*   **優先級**：P0 (最高)
*   **相依關係**：階段 1 (使用者角色驗證)
*   **狀態**：`[ ]`
*   **任務清單**：
    - [ ] **Task 2.1: 餐廳與餐點 API 開發**
      - *驗收條件*：開發 `GET /api/restaurants` 與 `GET /api/restaurants/{id}/menu`，回傳格式包含分類（Categories）與餐點（Meals）資訊。
    - [ ] **Task 2.2: 前端餐廳列表與菜單瀏覽頁面**
      - *驗收條件*：消費者登入後能瀏覽所有餐廳，點擊餐廳後能進入其菜單頁面，顯示分類與餐點價格。

### 階段 3：購物車 (Shopping Cart)
*   **優先級**：P0 (最高)
*   **相依關係**：階段 2 (需有餐點資料才能加入購物車)
*   **狀態**：`[ ]`
*   **任務清單**：
    - [ ] **Task 3.1: 前端購物車狀態管理實作**
      - *驗收條件*：在 React 中實作購物車 Context，提供加入購物車、修改數量、刪除項目功能，並即時計算小計。支援 LocalStorage 暫存。
    - [ ] **Task 3.2: 購物車資料合併 API (選填，視待確認事項釐清而定)**
      - *驗收條件*：登入後能同步前端購物車至後端，防止資料遺失。

### 階段 4：建立訂單 (Create Order)
*   **優先級**：P0 (最高)
*   **相依關係**：階段 3 (購物車內容作為訂單來源)
*   **狀態**：`[ ]`
*   **任務清單**：
    - [ ] **Task 4.1: 後端建立訂單 API**
      - *驗收條件*：`POST /api/orders` 能接收購物車明細，檢查付款方式（若外送則強制限刷卡）。寫入 `orders` 與 `order_items` 表，並扣除商品可用存量。
    - [ ] **Task 4.2: 模擬金流扣款邏輯 (信用卡)**
      - *驗收條件*：若選擇刷卡，後端需成功模擬金流扣款流程並回傳授權成功。
    - [ ] **Task 4.3: 前端結帳頁面實作**
      - *驗收條件*：消費者在購物車點擊結帳後，能選擇「到店自取（付現/刷卡）」或「外送（僅限刷卡）」，送出後產生訂單並跳轉至訂單狀態頁。

### 階段 5：查詢訂單狀態 (Order Tracking)
*   **優先級**：P1
*   **相依關係**：階段 4 (需先有訂單才能查詢)
*   **狀態**：`[ ]`
*   **任務清單**：
    - [ ] **Task 5.1: 訂單歷史與單一訂單狀態 API**
      - *驗收條件*：消費者可透過 `GET /api/orders` 查詢歷史紀錄，或以 `GET /api/orders/{id}` 獲取即時狀態。
    - [ ] **Task 5.2: 前端訂單狀態追蹤頁面**
      - *驗收條件*：提供輪詢（Polling）或 WebSocket 機制，即時向消費者展示訂單當前狀態（Created/Preparing/Delivering/Completed/Cancelled）。

### 階段 6：餐廳端更新訂單狀態 (Restaurant Order Management)
*   **優先級**：P1
*   **相依關係**：階段 4
*   **狀態**：`[ ]`
*   **任務清單**：
    - [ ] **Task 6.1: 餐廳端更新狀態 API**
      - *驗收條件*：餐廳角色可透過 `PATCH /api/orders/{id}/status` 將訂單更新為「準備中 (Preparing)」或「取消 (Cancelled)」。
    - [ ] **Task 6.2: 餐廳端接單管理後台頁面**
      - *驗收條件*：餐廳能即時看到新進訂單列表，點選「確認接單」或「拒絕接單（取消）」。

### 階段 7：管理員管理餐廳與餐點 (Admin Management)
*   **優先級**：P2
*   **相依關係**：階段 1、2
*   **狀態**：`[ ]`
*   **任務清單**：
    - [ ] **Task 7.1: 管理者管理 API 與營收統計 API**
      - *驗收條件*：提供帳戶啟用/停用 API，以及 `GET /api/admin/revenue` 獲取各餐廳營收統計數據。
    - [ ] **Task 7.2: 管理者後台控制面板**
      - *驗收條件*：管理者可檢視餐廳名冊、調整帳戶權限、查看全站與單店營收數據。
