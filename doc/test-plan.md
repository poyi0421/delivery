# 測試規格文件 (Test Plan) - 外送系統 (Delivery System)

## 1. 測試目標、範圍與排除項目
- **測試目標**：驗證本外送媒合平台在三端角色（消費者、店家、外送員）之間的互動流程與狀態轉移完全符合 PRD，確保無重大邏輯漏洞，且可在 SQLite 與 PostgreSQL 環境下穩定運行。
- **測試範圍**：
  - 用戶管理與認證（消費者、店家、外送員註冊與登入）。
  - 店家商品與菜單管理（CRUD）。
  - 消費者瀏覽店家、選餐與下單（貨到付款 COD，固定外送費 39 元，單店限制）。
  - 店家接單與標記待取餐。
  - 系統自動派單邏輯（指派給空閒時間最長之外送員）。
  - 外送員完成配送並收取現金。
- **排除項目**：
  - 線上金流信用卡扣款測試（本版本僅限貨到付款）。
  - 即時聊天與通訊功能測試（本版本無此功能）。
  - 外送員拒單測試（外送員無法拒單）。
  - 地圖 GPS 定位與即時行蹤軌跡測試（本版本不包含即時定位）。
  - 訂單取消與退款測試（已確認本系統完全不支援任何取消與退款）。
  - 外送員上下線切換與休息測試（外送員常駐上線，無下線功能）。

---

## 2. 測試策略
- **單元測試 (Unit Test)**：針對密碼雜湊雜湊邏輯、Pydantic 欄位驗證格式、訂單金額計算與外送員排序算法進行獨立測試。
- **整合測試 (Integration Test)**：驗證 SQLAlchemy ORM 對資料庫表關聯的寫入與讀取，特別是外鍵約束與軟刪除（將 `is_available` 設為 `False`）的級聯行為。
- **API 測試 (API Test)**：使用 FastAPI 提供的 `TestClient` 針對所有路由的 HTTP 狀態碼（200/201, 401, 404, 422, 500）與回應格式進行自動化測試。
- **前端元件測試 (Frontend Component Test)**：驗證 React 元件的表單驗證、購物車狀態更新與登入憑證的存取。
- **端對端測試 (E2E Test)**：手動或自動化地模擬消費者下單、店家接單並準備完成、系統指派、外送員送達的完整業務閉環流程。
- **手動驗收測試 (Manual Acceptance Test)**：部署至 Render.com 後，在不同瀏覽器與裝置（含手機版）上進行流程人工校對。

## 3. 測試環境與測試資料規則
- **本機開發與測試環境 (SQLite)**：
  - 本機測試執行時，使用記憶體資料庫 (`sqlite:///:memory:`) 或每次測試前清空本地的 `test.db`，確保測試資料互不干擾。
  - 測試結束後資料庫自動清空或 Rollback 事務。
- **正式與預備發佈環境 (Render PostgreSQL)**：
  - 測試資料必須與真實資料隔離，所有測試用使用者電子信箱一律使用 `test_` 前綴（如：`test_driver@example.com`）。
  - 測試完成後需手動或批次腳本清理含有測試前綴的資料。

## 5. 技術專題測試重點

### FastAPI 後端 API 測試重點
- **成功情境 (200/201 OK)**：輸入資料格式完整，API 回傳符合預期 JSON Schema。
- **資料驗證失敗 (422 Unprocessable Entity)**：傳入非法的 Email、價格小於 0、餐點數量小於等於 0，後端 Pydantic 應成功攔截並回傳錯誤明細。
- **身分未授權 (401 Unauthorized)**：未在 Header 攜帶 Authorization JWT Token，或 Token 已過期，API 應回傳 401 錯誤。
- **找不到資源 (404 Not Found)**：對不存在的商品 ID、訂單 ID 發送操作時，必須正確回傳 404 錯誤。
- **伺服器錯誤 (500 Internal Server Error)**：模擬資料庫連線中斷或伺服器非預期崩潰，驗證系統能回傳標準的 JSON 格式錯誤訊息而非回傳原始 Traceback。

### React 前端測試重點
- **畫面呈現**：消費者首頁需依狀態正確渲染商家列表；購物車內金額需正確加總；各後台面板僅能顯示該身分之操作按鈕。
- **使用者互動**：點擊「加到購物車」、「接單」、「確認送達」按鈕，確認能發送對應 API 並更新頁面狀態。
- **表單驗證**：註冊密碼欄位長度低於限制、Email 格式有誤時，前端需即時顯示紅字提示，並禁用送出按鈕。
- **載入與錯誤狀態**：在等待 API 回傳期間需顯示 Loading（載入中）遮罩；當網路斷線或 API 回傳 500 錯誤時，需彈出友善的錯誤通知。
- **API 串接**：利用 Axios 串接 FastAPI，確認對應 Token 已妥善保存於 LocalStorage 並在每次請求發送時自動夾帶。

### 資料庫測試重點
- **欄位限制**：測試當嘗試寫入重複 `email` 的 `users` 記錄、或對 `products.price` 寫入負數時，資料庫約束條件應能確實拒絕寫入。
- **關聯完整性**：外鍵關聯（如：`orders` 關聯 `users` 與 `merchants`）是否正常作用。
- **資料驗證與軟刪除**：驗證當商品被店家軟刪除時（`is_available = False`），消費者依然能從歷史訂單明細 `order_items` 中查詢到該餐點的快照售價。
- **遷移與索引**：執行 Alembic 遷移，確認 `idx_driver_dispatch` (role, driver_status, last_idle_at) 索引能正確套用在正式資料庫中。

### 安全性測試
- **機密資料保護**：
  - 檢查前端畫面的密碼輸入框為 `type="password"`（不顯示明文）。
  - 確認 `.env` 檔案未被 Git 追蹤，且線上主機的 API Key 無洩露。
- **權限控制 (RBAC)**：
  - 測試使用「消費者」的 Token 呼叫店家接單 API (`/accept`) 或外送員完成 API (`/complete`)，系統必須拒絕並回傳 403 Forbidden 狀態。
- **輸入防禦**：
  - 對 API 輸入參數進行 SQL 注入與 XSS 語法測試，確保 Pydantic 及 ORM 框架有對特殊字元進行防禦。
